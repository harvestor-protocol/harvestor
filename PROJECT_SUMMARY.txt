PROJECT OVERVIEW
════════════════════════════════════════════════════════════════════════════════

Harvestor brings financial inclusion to smallholder agriculture through 
on-chain credit scoring. This package contains the foundational score-attestation
smart contract for recording, verifying, and managing credit assessments on the
Stellar blockchain.

The contract enables authorized organizations (farming cooperatives, credit
assessment agencies) to submit cryptographically-backed credit scores for
farmers, with complete historical tracking for transparency and auditability.


KEY FEATURES
════════════════════════════════════════════════════════════════════════════════

✅ Score Submission
   - Authorized orgs submit credit scores (0-100) with evidence hashes
   - Cryptographic verification via require_auth()

✅ Access Control  
   - Admin functions: authorize/revoke submitter organizations
   - Submitter functions: submit scores for farmers
   - Role-based with signature verification

✅ Complete History
   - Immutable append-only ledger of all submissions
   - Chronologically ordered by ledger timestamp
   - Enables trend analysis, auditing, dispute resolution

✅ Optimized Storage
   - O(1) queries for latest score (credit decisions)
   - Per-farmer storage enables millions of farmers
   - Estimated cost: ~$100/month for 1M farmers

✅ Evidence Linking
   - Each score includes SHA-256 hash of off-chain evidence
   - Keeps on-chain footprint small
   - Enables external verification while preserving privacy

✅ Security Hardened
   - No unsafe Rust code
   - All inputs validated
   - Immutable records prevent tampering
   - Zero external dependencies

CONTRACT FUNCTIONS
════════════════════════════════════════════════════════════════════════════════

Admin Functions:
  authorize_submitter(admin, org)
    → Whitelist organization as score submitter
    
  revoke_submitter(admin, org)  
    → Remove organization from whitelist

Submission Function:
  submit_score(submitter, farmer, score, evidence_hash)
    → Attest credit score with evidence hash
    → Score must be 0-100
    → Updates latest + history

Query Functions:
  get_score(farmer) → Option<ScoreRecord>
    → Get farmer's most recent score (O(1) lookup)
    
  get_score_history(farmer) → Vec<ScoreRecord>
    → Get complete history (ordered by timestamp)


DATA STRUCTURE
════════════════════════════════════════════════════════════════════════════════

ScoreRecord:
  ├─ farmer: Address              // Farmer's Stellar address
  ├─ score: u32                   // Credit score (0-100)
  ├─ evidence_hash: BytesN<32>    // SHA-256 hash of evidence
  ├─ submitter: Address           // Organization that submitted
  └─ timestamp: u64               // UNIX timestamp from ledger

Immutable after creation → Prevents tampering, ensures auditability


STORAGE DESIGN
════════════════════════════════════════════════════════════════════════════════

Three-Layer Architecture:

1. SUBMITTERS (Whitelist)
   Key: Symbol("subs")
   Value: Vec<Address>
   Purpose: Authorized organizations
   Size: Typically <100 entries

2. LATEST_SCORE (Hot Path)
   Key: (&Symbol("lat"), farmer_address)
   Value: ScoreRecord
   Purpose: Fast O(1) credit lookups
   Size: 1 per farmer (~65 bytes)

3. SCORE_HISTORY (Audit Trail)
   Key: (&Symbol("hist"), farmer_address)  
   Value: Vec<ScoreRecord>
   Purpose: Complete immutable ledger
   Size: Grows with submissions (~2-10/farmer/year)

Cost Estimate (1 Million Farmers):
  Latest scores:        65 MB
  Score histories:     130 MB
  Authorized submitters: 3 KB
  ─────────────────────────
  Total:              195 MB
  Monthly cost:       ~$100 USD


SECURITY SUMMARY
════════════════════════════════════════════════════════════════════════════════

Access Control:
  ✅ All sensitive operations require signature verification
  ✅ Admin functions guarded by admin.require_auth()
  ✅ Submitter functions guarded by submitter.require_auth() + whitelist
  ✅ Query functions public (read-only)

Input Validation:
  ✅ Score range: [0, 100] enforced
  ✅ Addresses: Stellar type system validates
  ✅ Submitter: Whitelist check on each submission
  ✅ Evidence: Fixed 32 bytes (SHA-256 compatible)

Immutability:
  ✅ Scores append-only (never deleted)
  ✅ Records never modified
  ✅ Ledger-provided timestamps
  ✅ Cryptographic finality via Stellar consensus

Threat Model Addressed:
  ✅ Unauthorized submissions    → require_auth() + whitelist
  ✅ Admin impersonation         → Cryptographic signature
  ✅ Invalid scores              → Range validation
  ✅ Record tampering            → Immutable design
  ✅ Submitter spam              → Pre-approved organizations
  ✅ Ledger manipulation         → Stellar consensus

USE CASES
════════════════════════════════════════════════════════════════════════════════

1. Farmer Credit Application
   - Farmer applies for micro-loan
   - Lender queries contract: get_score(farmer)
   - Uses score to make lending decision

2. Score Update
   - Cooperative re-evaluates farmer
   - Submits new score: submit_score(...)
   - History automatically tracked

3. Auditing & Compliance
   - Regulator queries history: get_score_history(farmer)
   - Views complete submission timeline
   - Verifies immutability

4. Trend Analysis
   - Dashboard queries history
   - Shows farmer credit improvement over time
   - Identifies patterns

5. Dispute Resolution
   - Farmer questions score
   - Third party views complete history
   - Evidence hashes enable external verification
